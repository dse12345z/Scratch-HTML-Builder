從會用 AI，到會教 Agent 照你的流程做事

你會學會三件事：先分清楚 Agent / Prompt / Rule / Skill，再理解 SDD 為什麼能減少 AI 亂猜， 最後把 SDD 寫成一個 Codex Skill。每一課都有例子、測驗和可直接複製的 prompt。

學完你要能做到 這不是背名詞，是能拿來工作。
1判斷一個任務該用 prompt、rule 還是 skill。
2把模糊需求改寫成 spec。
3用 specify → plan → tasks → implement 控制 Codex。
4寫出一份可重複使用的 SDD Skill。
1

先分清楚：Agent、Prompt、Rule、Skill

這是最重要的地基。你不是要記名詞，而是要知道每種東西該放哪一層。

你先這樣理解

你平常跟 AI 說「幫我做登入功能」，這是 prompt。它只告訴 AI 這一次要做什麼，但沒有告訴 AI 應該怎麼工作。

所以 AI 可能直接改 code，也可能先問問題，也可能亂猜你的架構。這不是 AI 壞掉，而是你只給了任務，沒有給流程。

Prompt單次指令。回答：這次要做什麼？
Rule長期規則。回答：永遠不能違反什麼？
Skill可重複流程。回答：遇到這類任務時怎麼做？
Agent執行者。會讀檔、改 code、跑工具、回報結果。
一句話記住：Prompt 叫 AI 做事，Rule 約束 AI，Skill 教 AI 怎麼做事，Agent 負責執行。

不穩定的說法

幫我新增怪獸收藏功能。

問題是 AI 不知道要不要先看專案、要不要寫 spec、能改哪些檔案、做完要不要 build。

比較穩定的說法

請用 SDD 流程協助我新增怪獸收藏功能。
先不要改 code。
請先釐清需求並產出 spec，等我確認後再 plan、tasks、implement。

這句話開始規定流程，所以 Agent 比較不會爆衝。

小測驗：哪一個是 Skill？

點一個答案，看看你有沒有抓到。

正解是第三個。因為它描述的是「遇到 bug fix 這類任務時的固定流程」。
2

什麼工作適合做成 Skill？

不是所有任務都要 skill。Skill 是拿來處理重複、容易漏、需要流程的工作。

判斷公式

如果一個工作同時符合這幾個條件，就很適合做成 Skill：

  • 會重複發生，例如每週都要串 API、review PR、根據 Trello 卡實作。
  • 有固定流程，例如先讀規格、再找架構、再實作、再驗證。
  • 需要專案知識，例如 APIManager、ViewModel、Coordinator、coding style。
  • 容易做錯，例如 AI 常漏 fake data、漏錯誤狀態、亂改不相關檔案。
  • 有明確輸出，例如 code、spec、review report、測試結果。
記住這個公式：重複 + 有流程 + 容易漏 + 有標準輸出 = 適合做 Skill。

不適合做 Skill

  • 改 typo。
  • 把一個顏色改掉。
  • 問某段 code 是什麼意思。

這些任務太小，用 prompt 就好。

適合做 Skill

  • 根據 API 文件新增一支 API。
  • 依團隊規範 review Swift PR。
  • 做一個需要 spec、plan、tasks 的新 feature。

這些任務有流程、有標準、也容易漏步驟。

你可以這樣問自己

  1. 這件事我是不是做過很多次？
  2. 如果交給新人，我會不會寫一份 SOP？
  3. AI 做這件事時，最常漏哪一步？
  4. 最後結果有沒有固定格式？

如果答案大多是「有」，它就是 skill 候選人。

小測驗：哪個最適合做成 Skill？

正解是第二個。它重複、有流程、需要專案知識，而且有明確輸出。
3

SDD 基礎：為什麼不要直接叫 AI 寫 code？

AI 通常不是不會寫 code，而是不知道你真正要什麼。

SDD 是什麼

SDD 是 Spec Driven Development。簡單講：先把「要做什麼」定清楚，再決定「怎麼做」，最後才寫 code。

你說「新增怪獸收藏功能」，AI 還不知道收藏狀態存本機還是 server、列表和詳情要不要同步、沒登入能不能收藏、失敗要怎麼顯示。這些都不講，它就只能猜。

Spec定義需求：使用者可以做什麼、畫面有哪些狀態、什麼算完成。
Plan定義技術方案：改哪些檔案、資料怎麼流、有什麼風險。
Tasks定義步驟：第一步做什麼、每一步如何驗證。
Code最後才實作，而且要對照 spec 驗證。

Spec 應該回答什麼

  • 使用者故事：誰，在什麼情境，要完成什麼事？
  • 功能範圍：這次要做哪些行為？
  • 不做事項：搜尋、排序、同步、登入限制這次做不做？
  • UI 狀態：載入、空資料、錯誤、成功、已收藏、未收藏。
  • 驗收條件：怎樣算完成？

模糊需求

我想做一個怪獸圖鑑功能。

改成 Spec 問題

入口在哪？
列表顯示哪些欄位？
詳情頁顯示哪些資料？
載入失敗怎麼辦？
空資料怎麼辦？
這次不做搜尋和排序嗎？
怎樣算完成？
請不要寫 code。
請先把以下需求整理成 SDD spec：

需求：新增怪獸收藏功能

請包含：
- user stories
- in scope
- out of scope
- UI states
- data states
- acceptance criteria

小測驗：Spec 主要在定義什麼？

對。Spec 先定義行為，不急著談 implementation detail。
4

SpecKit 工作流：specify → plan → tasks → implement

這一課教你怎麼實際控制 Codex，不讓它一開始就衝去改 code。

1. specify

只釐清需求。這一步不讀太多 code，也不改檔案。目標是讓需求沒有大洞。

2. plan

讓 Codex 讀專案，提出技術方案。這一步開始談 affected files、data flow、risks、verification。

3. tasks

把 plan 拆成小步驟。每個 task 都要有 done criteria，不然 Agent 會一次做太多。

4. implement

最後才實作。一次做一個 task，完成後驗證，並回頭檢查是否符合 spec。

你控制 Codex 的關鍵，不是叫它「小心一點」，而是把它每一步允許做的事切清楚。

Plan Prompt

請根據剛剛的 spec 讀取目前專案，產出 implementation plan。
先不要修改檔案。

請包含：
- affected files
- data flow
- UI changes
- risks
- verification method

Tasks Prompt

請把 plan 拆成 tasks。
每個 task 請包含：
- goal
- files likely touched
- done criteria
- verification
請依相依性排序。

Implement Prompt

請先實作 Task 1。
只修改 Task 1 需要的檔案。
完成後請回報：
- 修改了哪些檔案
- 如何驗證
- 是否符合 spec

Constitution / Governance 是什麼

Constitution 是專案不可違反的原則，例如不要繞過 APIManager、不要在 ViewController 塞商業邏輯、不要新增未經同意的套件。

Governance 是規格怎麼維護，例如需求變更先改 spec、實作完成要更新 tasks、review 時要檢查 code 是否符合 spec。

小測驗：哪個順序比較穩？

對。先定需求，再定方案，再拆任務，最後才實作。
5

把 SDD 變成 Agent Skill

現在把你學到的流程封裝起來，讓 Codex 以後每次都照這套方法做。

Skill 不是更長的 prompt

Skill 是「遇到某類任務時，Agent 應該採用的工作方法」。它要寫清楚何時使用、輸入是什麼、步驟怎麼走、什麼時候要停下確認、最後要輸出什麼。

When to use什麼任務觸發這個 skill。
WorkflowAgent 要照哪個順序工作。
Checkpoints何時不能繼續、何時要驗證。
Output最後回報格式要穩定。

一份可用的 SDD Skill

# SDD Feature Development Skill

Use this skill when implementing a non-trivial feature that changes user-visible behavior, touches multiple files, or requires technical decisions.

## Workflow

1. Clarify the requirement.
2. Write or update the spec.
3. Ask for confirmation if the spec has unresolved questions.
4. Read the codebase and create an implementation plan.
5. Break the plan into tasks.
6. Implement one task at a time.
7. Verify with tests, build, or manual checks.
8. Review the implementation against the spec.
9. Summarize changes, verification, and risks.

## Checkpoints

- Do not implement before the spec is clear.
- Do not modify unrelated files.
- Do not skip plan and tasks for non-trivial features.
- Ask before making broad architectural changes.
- Always verify after implementation.

## Final Output

- What changed
- Files touched
- Verification result
- Remaining risks

完整實戰 Prompt

當你要用 Codex 跑一個新 feature，可以直接貼這段：

請使用 SDD Feature Development Skill 協助我做 iOS feature。
Feature：新增怪獸收藏功能。

規則：
1. 先不要改 code。
2. 先產出 spec，包含 user stories、scope、out of scope、states、acceptance criteria。
3. 等我確認後，再讀專案並產 plan。
4. plan 必須列 affected files、data flow、risks、verification。
5. 再拆 tasks，每個 task 都要有 done criteria。
6. 實作時一次只做一個 task。
7. 完成後執行 build 或說明無法執行的原因，最後做 spec 對照 review。

小測驗：Skill 最重要的是什麼？

對。Skill 的價值在穩定流程，不在漂亮語氣。
你現在學到的核心：SDD 是流程，Skill 是把流程交給 Agent 穩定執行的方法。

練習：產生你自己的 SDD Prompt

輸入你想做的功能，網頁會幫你組出一段可以貼給 Codex 的 SDD 起手式。

先用這段 prompt 讓 Codex 寫 spec，不要急著改 code。
